2. RyanCodrai/turbovec:
   star: 10373  语言/技术: Rust, Python, SIMD (NEON, AVX-512BW, AVX2), TurboQuant 量化算法, Lloyd-Max 标量量化, maturin
   项目简介: turbovec 是一个基于 Rust 实现的高性能向量索引库，提供 Python 绑定，核心采用 Google Research 的 TurboQuant 算法。它解决了大规模向量检索中内存占用高、需要训练阶段、搜索速度慢等痛点——能将 1000 万文档的内存从 31 GB（float32）压缩到仅 4 GB，同时搜索速度比 FAISS 更快。支持在线增量添加、搜索时过滤、稳定 ID 映射等特性，适用于对隐私、内存或延迟敏感的 RAG 场景。
   亮点: 极致压缩：1000 万文档从 31 GB 压缩到 4 GB（16x 压缩比）; 搜索速度超越 FAISS：ARM 上快 12–20%，x86 上持平或更优; 无需训练阶段：在线增量添加向量，无参数调优，无需重建索引; 搜索时过滤：支持 ID 白名单或位掩码过滤，在 SIMD 内核层面短路跳过无关块，无召回损失; 与主流框架集成：提供 LangChain、LlamaIndex、Haystack、Agno 的即插即用替换; 纯本地运行：无需托管服务，数据不出机器，支持完全离线的 RAG 栈
   详细报告: turbovec 是一个用 Rust 编写的高性能向量索引库，通过 Python 绑定让 Python 用户也能轻松使用。它的核心技术基于 Google Research 发表的 TurboQuant 算法（ICLR 2026），这是一种数据无关的量化器，能达到 Shannon 失真率下界的理论最优水平，且不需要码本训练和单独的训练阶段。

**核心工作原理**分为六个步骤：
1. **归一化**：剥离向量长度（模），将每个向量转化为单位方向向量，模单独存储为单个浮点数。
2. **随机旋转**：将所有向量乘以同一个随机正交矩阵，旋转后每个坐标独立服从 Beta 分布（高维下收敛于高斯分布），这使得坐标分布可预测且与输入数据无关。
3. **逐坐标校准（TQ+）**：在首次添加时，为每个坐标拟合平移和缩放两个标量，将经验分位数映射到标准 Beta 边际分布上。校准在首次 add 后冻结，后续添加复用，无需重新训练。这一步在低维嵌入（如 GloVe 2-bit）上可带来最多 +1.4pp 的召回提升。
4. **Lloyd-Max 标量量化**：由于分布已知，可预先计算最优分桶方案。2-bit 对应 4 个桶，4-bit 对应 16 个桶。这些桶边界和质心从数学计算得出，而非从数据学习。
5. **位打包**：每个坐标变为小整数（0-3 或 0-15），紧密打包成字节。1536 维向量从 6144 字节压缩到 384 字节，实现 16 倍压缩。
6. **长度重归一化评分**：标量量化会系统性低估内积。编码时计算每个向量的校正标量（原向量与其重建之间的内积比），搜索内核在堆插入前乘以该校正因子，使估计量无偏，零搜索开销，零额外存储。

**搜索过程**：查询向量旋转到同一空间后，直接对码本值评分，无需解压数据库向量。评分内核使用手写 SIMD 指令（ARM 用 NEON，x86 用 AVX-512BW，回退到 AVX2），采用 nibble-split 查找表实现最大吞吐量。

**关键特性**：
- **在线增量添加**：向量添加即索引，无训练步骤，无参数调优，索引随语料增长无需重建。
- **搜索时过滤**：`search()` 方法接受 ID 白名单或槽位位掩码，在内核层面以 32 向量块粒度进行短路优化——无允许槽位的块直接跳过，块内非允许槽位在堆插入时丢弃。选择性过滤（白名单远小于索引）可避免大部分 SIMD 计算开销。输出长度为 min(k, len(allowed))，不会用填充结果凑数。
- **稳定 ID（IdMapIndex）**：支持外部 uint64 ID 映射，删除操作 O(1)，ID 在持久化后仍然保留。
- **混合检索**：可与 SQL、BM25、ACL、时间窗口等外部系统配合，先由外部系统缩小候选集，再由 turbovec 进行稠密重排序。
- **框架集成**：提供 LangChain、LlamaIndex、Haystack、Agno 的即插即用替换，相同接口、相同的持久化和检索器管线，只需更换 import 即可。

**性能表现**：在 10 万向量、1000 查询、k=64 的基准测试中，ARM（Apple M3 Max）上比 FAISS IndexPQFastScan 快 12–20%；x86（Intel Xeon Sapphire Rapids）上 4-bit 配置领先 1–6%，2-bit 单线程差距在 1% 以内。召回率方面，在 OpenAI d=1536 和 d=3072 上，TurboQuant 在 R@1 上比 FAISS 高 0.4–3.4 个百分点（2-bit 和 4-bit），k=4 时两者均收敛到 1.0。

**适用场景**：构建对隐私、内存或延迟有严格要求的 RAG 系统，配合任意开源嵌入模型可实现完全离线的 RAG 栈。

   项目简介: 一句话概括：whichllm 是一个帮你"看菜下饭"的本地大模型推荐神器。它能自动检测你的电脑硬件（GPU/CPU/RAM），然后从 HuggingFace 海量模型中，基于真实跑分（而非单纯看模型大小）排名出最适合你机器的那款 LLM。它解决了"我的显卡能跑哪个模型？哪个跑得最快最好？"这个让无数人头疼的痛点，堪称本地部署大模型的"装机指南针"。

   详细报告: whichllm 的核心价值在于它不是简单地回答"你的显存能装下哪个模型"，而是回答"能装下的模型里，哪个实际表现最好"。这个区别至关重要——一个 32B 模型可能刚好塞进 24GB 显卡，但一个更新的 27B 模型可能跑分更高、速度更快，whichllm 会正确地把 27B 排第一。

项目的技术壁垒主要体现在评分引擎上：它整合了 LiveBench、Artificial Analysis Index、Aider、Chatbot Arena ELO、Open LLM Leaderboard 等多个基准来源，并对每个分数进行证据置信度加权（直接匹配满分，自报分数打 55 折）。同时，它还会对过时榜单进行"年代降级"处理，防止 2024 年的老模型因为旧榜单的高分而压制 2025 年的新模型。
